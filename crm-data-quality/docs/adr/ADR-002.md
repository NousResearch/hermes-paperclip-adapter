# ADR-002: Integration API Surface Pattern

## Status
Accepted

## Context
The integration layer needs a stable API surface for extraction, scoring, and status write-back. Directly coupling UI or workflow callers to low-level Dataverse endpoints would create brittle dependencies.

## Decision
Adopt a bounded internal API surface with three operations:
- `extract` for canonical entity snapshots
- `score` for rule execution and quality scoring
- `publish` for persisting scores/remediation metadata

## Consequences
- Positive: Clear separation between ingestion, processing, and persistence.
- Positive: Enables independent iteration of scoring engine.
- Negative: Requires adapter code for Dataverse specifics.
- Negative: Additional contract maintenance overhead.

## Alternatives Considered
- Pass-through raw Dataverse APIs: rejected due to tight coupling and weak portability.
- Single monolithic endpoint: rejected due to poor observability and harder retries.
