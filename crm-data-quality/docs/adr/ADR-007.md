# ADR-007: Error Handling and Retry Semantics

## Status
Accepted

## Context
Dataverse operations can fail transiently (throttling/network) or permanently (validation/authorization). A single retry policy for all failures causes either wasted load or data loss.

## Decision
Classify failures into transient and terminal categories; apply bounded exponential backoff retries only to transient classes and dead-letter terminal failures with diagnostics.

## Consequences
- Positive: Higher reliability under transient faults.
- Positive: Better observability for unrecoverable errors.
- Negative: Needs curated error taxonomy and telemetry.
- Negative: Dead-letter processing introduces operational workflow.

## Alternatives Considered
- Retry everything uniformly: rejected due to noisy failures and wasted capacity.
- Fail fast with no retry: rejected because transient faults are expected in cloud integrations.
