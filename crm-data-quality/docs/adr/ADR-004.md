# ADR-004: Sync Model and Change Detection Strategy

## Status
Accepted

## Context
Full-table pulls are expensive and can miss near-real-time remediation targets. The integration needs a sync model balancing freshness, API limits, and recoverability.

## Decision
Implement incremental sync using high-watermark timestamps with periodic reconciliation backfills.

## Consequences
- Positive: Lower API consumption than repeated full extracts.
- Positive: Better latency for data-quality insights.
- Negative: Requires robust watermark state handling.
- Negative: Late-arriving updates need reconciliation policy.

## Alternatives Considered
- Full snapshot every run: rejected due to scale and cost.
- Event-only sync with no backfill: rejected due to missed-event recovery gaps.
